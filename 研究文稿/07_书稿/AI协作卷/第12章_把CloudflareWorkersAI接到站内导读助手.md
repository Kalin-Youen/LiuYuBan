# 第12章 把 Cloudflare Workers AI 接到站内导读助手

如果这个站点要先做一个“站内实时问答 / 导读助手”，那最顺手的起点确实就是 `Cloudflare Workers AI`。

原因不是它在一切意义上都最强，而是它和现在这个项目的站点形态最匹配：

1. 这是一个静态站点，最怕的是为了接一个问答框，先背上一整套后端。
2. Cloudflare Worker 本身就适合做“轻后端代理”，可以把模型调用、跨域、限流和后续的升级都包在一层里。
3. 它天然适合接流式返回，读者不会觉得自己在点一个死按钮等整段文字突然出现。
4. 后面如果要换模型、加日志、加缓存、加向量检索，也是在同一条线上继续长，而不是推倒重来。

所以这里更推荐的路线，不是“把 API key 直接塞进前端”，而是：

`静态站前台 -> Cloudflare Worker -> Workers AI`

这条线的关键价值，是把**站点入口、上下文筛选、模型调用和后续治理**拆开。

## 一、先把边界说清

这个导读助手的工作，不应该被理解成“替整套书下最终结论”。

它更合适的职责只有三层：

1. 先帮读者在站内找到最相关的章节。
2. 再把这些章节压成一条最短可走通的阅读线。
3. 最后用较白话的方式解释，为什么要先读这里，再读那里。

也就是说，它首先是一个**站内导读器**，其次才是一个**问答器**。

如果这一点不先定住，很容易出现两个偏移：

1. 它开始脱离站内文本，自顾自讲一套外部理论。
2. 它一上来就替作者说“这就是最终答案”，把整个项目又说回强立场。

因此，真正稳定的做法不是只让模型看用户提问，而是同时给它：

1. 站点主线说明
2. 当前最相关的章节摘录
3. 明确的回答纪律

## 二、最小可行接法

当前站点已经有 `content.json`，这件事非常关键。

因为它意味着我们不必一开始就上数据库、向量库或复杂检索系统，而是可以先用站内已有索引做一层本地筛选：

1. 读者在前台输入问题
2. 前台从 `content.json` 里筛出最相关的 3 到 4 篇章节
3. 把这些章节的标题、卷册、摘录和站内链接一起发给 Worker
4. Worker 再把这些上下文交给 Workers AI 生成回答

这样做的好处是：

1. 架构轻
2. 成本低
3. 很适合先上线再迭代
4. 回答天然会更贴近站内现有内容

这一版先不要急着追求“大而全”的 RAG。

先让它做到：

1. 能回答“这个概念在哪些章节谈过”
2. 能回答“第一次该从哪篇进入”
3. 能回答“生活卷和主书稿之间怎么互相接”
4. 能把回答里的章节卡片点开跳转

只要这四件事站稳，它就已经是一个合格的站内导读助手了。

## 三、为什么 Worker 这一层不能省

很多人一开始会想：既然是静态站，那能不能直接在前端请求模型？

不建议。

原因很直接：

1. 密钥不能直接暴露在前端。
2. 后面一旦要加限流、改模型、控费用，就没有缓冲层。
3. 你未来很可能还要在这一层加访问日志、来源校验、敏感问题过滤和回答格式控制。

所以 Worker 这一层不是多余的技术动作，而是后续治理能力的开始。

从活书系统的角度看，这一层就像评论系统里的“回写协议”一样：

它不是为了复杂，而是为了让后续修改机制比内容本身更稳定。

## 四、这版站内导读助手建议先守的纪律

为了避免它越用越飘，建议先守下面几条：

1. 优先引用站内上下文，不够再明确承认不够。
2. 优先给阅读线，不要先给宏大判断。
3. 优先白话解释，不要先抬高理论姿态。
4. 优先指出相关章节，不要把回答做成无出处的长段落。
5. 优先把“较稳的已写内容”和“尚在候写的方向”区分开。

这五条，其实就是把“导读”放在“作答”前面。

## 五、推荐的部署顺序

实际接入时，可以按这个顺序推进：

1. 在 Cloudflare 新建 Worker
2. 给 Worker 绑定 `AI`
3. 把前台请求统一发到这个 Worker 地址
4. 先只接站内章节摘录，不上额外数据库
5. 前台先做流式显示和章节卡片跳转
6. 跑通之后，再考虑缓存、来源限制、日志与更强检索

这条顺序的重点是：**先把闭环打通，再补性能与治理。**

## 六、当前仓库里对应的落点

这一版已经把几个关键落点预留出来了：

1. `site.config.json`
   负责放导读助手的入口文案、Worker 地址、默认模型和提示语。
2. `docs/assets/app.js`
   负责前台问题提交、本地章节筛选、流式显示和章节跳转。
3. `workers/cloudflare-guide-assistant.js`
   负责 Worker 代理示例。

所以这章的作用，不是重复讲一遍“AI 很厉害”，而是把这件事先压成一个**可直接接线的站内接口方案**。

## 七、后面如果继续长，优先长什么

如果这一版站稳了，下一步最值得继续补的不是花哨能力，而是下面这些：

1. 为不同问题类型分流
   例如“概念在哪”“从哪开始读”“相关章节怎么串”“哪里还没写稳”。
2. 为回答增加来源卡
   让每次回答都能直接跳到章节。
3. 为高频问题做缓存
   减少重复调用成本。
4. 为图谱节点增加联动
   让“知识图谱 -> 导读助手 -> 章节”变成连续入口。
5. 为活书回写预留问题沉淀口
   把高频追问和长期卡点转回后续写作与补桥。

简化成一句话就是：

**先把它做成站内阅读接口，再把它慢慢长成活书系统的一部分。**
