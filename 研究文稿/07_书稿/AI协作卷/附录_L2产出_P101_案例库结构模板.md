## 案例库分类体系

案例库按五类场景组织：

| 场景类别 | 说明 | 典型问题 |
|---|---|---|
| 学习 | 从混乱输入到结构化掌握 | 学了很多但不能迁移；收藏很多但不能形成能力 |
| 写作 | 从观点堆积到论证边界 | 选题太散；观点容易写过强；AI输出很快但结构不稳 |
| AI 使用 | 从答案消费到协作判断 | AI输出很多但判断力没变强；不知道AI回答哪里可能错 |
| 项目推进 | 从忙乱耗散到结构增量 | 会议产生大量耗散；反馈没有沉淀；组织忙但不形成结构 |
| 赚钱实践 | 从空想机会到可验证交换 | 赚钱想法是不是只是空想；如何判断机会是否可用 |

## 单个案例模板

```markdown
## 案例编号：C-场景-序号（如 C-学习-01）

### 基本信息

| 字段 | 内容 |
|---|---|
| 场景类别 | 学习 / 写作 / AI使用 / 项目推进 / 赚钱实践 |
| 案例标题 | 一句话说明 |
| 提交者 | （匿名/昵称） |
| 提交日期 |  |
| 是否成功案例 | 成功 / 失败 / 混合 |

### 案例正文

| 字段 | 内容 |
|---|---|
| 原问题 | 遇到了什么问题？ |
| 原错误 | 不用方法时会怎么乱？ |
| 使用的方法卡/模板 | 用了哪张卡或哪个模板？ |
| 行动过程 | 具体做了什么？ |
| 结果变化 | 使用方法后差异在哪里？ |
| 下一步行动 | 接下来该做什么？ |
| 结果如何反馈 | 反馈从哪来？如何接收？ |

### 残差与边界

| 字段 | 内容 |
|---|---|
| 剩余残差 | 还有什么没解决？ |
| 方法失效点 | 方法在哪里不够用？ |
| 下次如何改 | 如果再来一次，会怎么调整？ |
| 是否需要新卡/新模板 | 现有工具是否覆盖了这个场景？ |

### 标签

| 字段 | 内容 |
|---|---|
| 差异轴 | 信息差/能力差/目标差/制度差/其他 |
| 耗散类型 | 正向/维持性/负向 |
| 结构沉淀 | 有/无/部分 |
| 吸收系数评估 | 高/中/低 |
```

## 失败案例特别标注

失败案例比成功案例更重要，因为失败案例能让方法长出边界。失败案例需额外填写：

```markdown
### 失败分析

| 字段 | 内容 |
|---|---|
| 失败类型 | 差异不够清晰 / 耦合机制不存在 / 释放路径成本过高 / 反馈太慢 / 封存失败 / 主体吸收系数太低 / 环境变化超过稳态窗口 / 对照模型解释力更强 |
| 失败是否可预见 | 是 / 否 |
| 失败是否可避免 | 是 / 否 |
| 失败后是否产生新结构 | 是 / 否 |
| 失败是否暴露了方法的边界 | 是 / 否 |
```

## 不确定点

1. 案例模板字段是否过多？实际用户可能不愿填写全部字段。
2. 失败案例的"失败类型"分类是否完整？
3. 案例库是否需要"轻量版"（只填5个核心字段）？

## 风险点

1. 案例可能被拔高为"方法有效的证明"，但案例只能说明"在某个场景下有用"。
2. 失败案例可能被回避，导致案例库偏向成功叙事。
3. 案例的真实性无法由 L2 验证，必须由作者或用户确认。

## 需要 L3/L4 复核的问题

1. 案例模板是否需要轻量版？——需 L3 裁定。
2. 失败类型分类是否完整？——需 L3 评估。
3. 案例库回写位置？——需 L4 裁定。
4. 案例素材来源？——需 L4 指定。
