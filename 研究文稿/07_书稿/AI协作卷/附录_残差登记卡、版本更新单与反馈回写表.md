# 附录 残差登记卡、版本更新单与反馈回写表

这一附录不是理论章节，而是供作者、AI 与读者在后续协作中直接使用的工作模板。它的目标很简单：让“残差、版本、反馈”三类最容易在扩写中丢失的东西，获得统一写法。

## 一、残差登记卡

适用情形：

- 某个问题反复出现但尚不能并入主线；
- 某段扩写必须降强，不能直接写成结论；
- 某类读者误解持续出现，但还没有专门接口处理。

最小模板：

```text
残差名称：
出现位置：
所属主线节点：
问题类型：
当前状态：
当前处理动作：
建议回写位置：
下一步条件：
关联章节 / 术语：
备注：
```

建议填写说明：

- `残差名称` 用一句短语命名，不要写成整段说明；
- `出现位置` 标出章节、专题、站点入口或反馈来源；
- `问题类型` 可选：术语漂移 / 推导缺桥 / 边界过宽 / 自我失真风险 / 反馈噪声 / 待验证接口；
- `当前状态` 可选：立即回看 / 本轮待处理 / 持续观察 / 可自然耗散；
- `当前处理动作` 可选：吸收 / 降强 / 转专题 / 暂挂 / 删除；
- `下一步条件` 必须写清楚它需要什么材料、证据或章节承接。

## 二、版本更新单

适用情形：

- 主线发生补桥、收紧或重写；
- 一个核心词改变了职责；
- 新增章节、专题或入口改变了整体阅读路径。

最小模板：

```text
更新名称：
更新时间：
变更类型：
涉及文件：
变更摘要：
旧写法失效点：
新写法保留点：
需要同步回看的位置：
关联残差：
入口 / 术语同步情况：
```

建议填写说明：

- `变更类型` 可选：主线变更 / 边界变更 / 接口变更 / 残差变更 / 入口变更；
- `变更摘要` 应压成一到三句，说明这次到底改了哪一层结构；
- `旧写法失效点` 用来防止系统继续沿旧主线扩写；
- `新写法保留点` 用来说明这次不是纯否定，而是重写后的新稳定点；
- `需要同步回看的位置` 至少写出术语表、总结章、白话卷或站点入口中受影响的部分。

## 三、读者反馈回写表

适用情形：

- 收到值得保留的读者反馈；
- 发现某一段虽然读懂了，但没有留下判断路径；
- 某个章节引发稳定误解或稳定影响。

最小模板：

```text
反馈来源：
读者类型：
命中章节：
反馈类型：
反馈内容压缩：
命中点判断：
是否进入判断：
建议动作：
回写位置：
是否新增残差 / 版本：
```

建议填写说明：

- `反馈类型` 可选：进入困难 / 误解点 / 无感区 / 高影响段 / 新接口需求；
- `反馈内容压缩` 要压成一到三句，不要原样堆评论；
- `命中点判断` 说明它到底打在定义、主线、边界、入口还是表达顺序上；
- `是否进入判断` 用来区分“读者只是看懂了”还是“读者真的开始调用这套尺度”；
- `建议动作` 可选：改导读 / 补桥 / 降强 / 转专题 / 登记残差。

## 四、最小使用节奏

如果不想把这三张表用得太重，可以采用一个最小节奏：

1. 每轮较大扩写后，至少补一次`版本更新单`；
2. 遇到不能立即吸收的问题，就补一张`残差登记卡`；
3. 收到值得保留的读者反馈，就先压成`反馈回写表`；
4. 每经过一轮较大重写，再统一回看一次三张表之间是否已经形成回路。

这套节奏的目标，不是增加管理动作，而是让重要变化不再只留在聊天记录和个人记忆里。

## 五、三张表的连接方式

这三张表不是并列孤立的。更稳的连接方式是：

1. 读者反馈先进入`反馈回写表`；
2. 若当前不能直接吸收，则转入`残差登记卡`；
3. 一旦正式改动结构，再生成`版本更新单`；
4. 版本更新后，再反过来修正导读、术语和站点入口。

也就是说：

> 反馈负责把问题带进来，残差负责让问题不丢失，版本单负责让改动留下结构痕迹。

## 六、这一附录的压缩结论

若把这份附录压成一句最实用的话，那就是：

> 凡不能立即并入主线的，就登记；凡已经改变结构的，就留版本；凡来自读者现实接收的，就先压缩成可回写反馈，而不是让它们散在对话和记忆里。
