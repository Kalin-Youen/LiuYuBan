# 附录 残差登记卡、版本更新单与反馈回写表

这一附录不是理论章节，而是供作者、AI 与读者在后续协作中直接使用的工作模板。它的目标很简单：让“随手评论、残差、版本、反馈、修改边界与触发判断”这几类最容易在扩写中丢失的东西，获得统一写法。

## 一、残差登记卡

适用情形：

- 某个问题反复出现但尚不能并入主线；
- 某段扩写必须降强，不能直接写成结论；
- 某类读者误解持续出现，但还没有专门接口处理。

最小模板：

```text
残差名称：
出现位置：
所属主线节点：
问题类型：
当前状态：
当前处理动作：
建议回写位置：
下一步条件：
关联章节 / 术语：
备注：
```

建议填写说明：

- `残差名称` 用一句短语命名，不要写成整段说明；
- `出现位置` 标出章节、专题、站点入口或反馈来源；
- `问题类型` 可选：术语漂移 / 推导缺桥 / 边界过宽 / 自我失真风险 / 反馈噪声 / 待验证接口；
- `当前状态` 可选：立即回看 / 本轮待处理 / 持续观察 / 可自然耗散；
- `当前处理动作` 可选：吸收 / 降强 / 转专题 / 暂挂 / 删除；
- `下一步条件` 必须写清楚它需要什么材料、证据或章节承接。

## 二、版本更新单

适用情形：

- 主线发生补桥、收紧或重写；
- 一个核心词改变了职责；
- 新增章节、专题或入口改变了整体阅读路径；
- 需要保证改坏后可回滚。

最小模板：

```text
更新名称：
更新时间：
父版本 / 快照编号：
变更类型：
涉及文件：
变更摘要：
旧写法失效点：
新写法保留点：
回滚锚点：
需要同步回看的位置：
读者可见说明：
入口 / 术语同步情况：
关联残差：
```

建议填写说明：

- `父版本 / 快照编号` 用来说明这次修改是从哪一版长出来的；
- `变更类型` 可选：主线变更 / 边界变更 / 接口变更 / 残差变更 / 入口变更；
- `变更摘要` 应压成一到三句，说明这次到底改了哪一层结构；
- `旧写法失效点` 用来防止系统继续沿旧主线扩写；
- `新写法保留点` 用来说明这次不是纯否定，而是重写后的新稳定点；
- `回滚锚点` 应写清楚旧稿位置、快照文件或版本标签；
- `读者可见说明` 至少要让常读者知道“哪里改了、为什么改、旧版是否仍可回看”；
- `需要同步回看的位置` 至少写出术语表、总结章、生活卷或站点入口中受影响的部分。

## 三、回写修改任务单

适用情形：

- 某条评论明显只打在局部段落，不应把整章一起重写；
- 准备让 AI 协助修改，但必须先锁定范围和幅度；
- 需要保护核心骨架不被顺手改散。

最小模板：

```text
任务名称：
触发来源：
命中章节 / 段落：
本轮目标：
允许修改范围：
禁止改动骨架：
改动上限：
必须携带的锚点材料：
是否需要先留快照：
完成后必须生成的附带对象：
备注：
```

建议填写说明：

- `触发来源` 标出对应评论、生长卡、反馈表或残差卡；
- `本轮目标` 只写这次最想解决的一个主问题，不要把所有不满一起塞进来；
- `允许修改范围` 尽量写成一句 / 一段 / 一小节 / 只补桥不重排；
- `禁止改动骨架` 写明哪些核心命名、章节顺序或论证链暂时不能碰；
- `改动上限` 可选：措辞修补 / 局部降强 / 局部补桥 / 小节级调整 / 需升级为版本级改写；
- `必须携带的锚点材料` 至少写清全书摘要、术语表、写作风格说明里哪些必须带上；
- `完成后必须生成的附带对象` 可选：版本单 / 公开变更说明 / 残差卡 / 入口同步单。

## 四、评论筛选与冲突归并表

适用情形：

- 评论很多，不能把每一条都直接喂给 AI；
- 多条评论看起来都在说问题，但其实质量差异很大；
- 彼此矛盾的评论同时出现，需要先分清再决定是否回写。

最小模板：

```text
评论来源：
原始评论：
质量判断：
命中问题：
是否与其他评论同向汇聚：
是否与其他评论形成冲突：
冲突组编号：
建议处理动作：
是否需要人工复核：
备注：
```

建议填写说明：

- `质量判断` 可选：建设性 / 待观察 / 灌水 / 恶意风险；
- `命中问题` 用一到两句压清，不要原样堆评论；
- `同向汇聚` 用来判断这是不是多个读者都在暴露同一缺口；
- `形成冲突` 用来防止 AI 把彼此相反的意见平均成一句模糊话；
- `建议处理动作` 可选：直接进入生长卡 / 合并同类 / 暂挂观察 / 忽略 / 人工复核；
- `是否需要人工复核` 在恶意风险高、问题牵涉主线、或冲突过强时应写“是”。

## 五、读者反馈回写表

适用情形：

- 收到值得保留的读者反馈；
- 发现某一段虽然读懂了，但没有留下判断路径；
- 某个章节引发稳定误解或稳定影响。

最小模板：

```text
反馈来源：
读者类型：
命中章节：
反馈类型：
反馈内容压缩：
命中点判断：
反馈权重：
是否进入判断：
建议动作：
回写位置：
是否新增残差 / 版本 / 公开说明：
```

建议填写说明：

- `反馈类型` 可选：进入困难 / 误解点 / 无感区 / 高影响段 / 新接口需求；
- `反馈内容压缩` 要压成一到三句，不要原样堆评论；
- `命中点判断` 说明它到底打在定义、主线、边界、入口还是表达顺序上；
- `反馈权重` 可选：低 / 中 / 高；高权重通常对应主线误解、反复卡点或现实接口强反馈；
- `是否进入判断` 用来区分“读者只是看懂了”还是“读者真的开始调用这套尺度”；
- `建议动作` 可选：改导读 / 补桥 / 降强 / 转专题 / 登记残差；
- `是否新增残差 / 版本 / 公开说明` 用来提醒这次反馈是否已经触到结构级修改。

## 六、独立评审处理单

适用情形：

- 收到来自外部 AI、独立读者或整站体验者的长篇评审；
- 一份评审里同时混着入口、语气、内容、站点和协作流程问题；
- 需要防止“有价值的批评”和“没读到就推断的判断”被平均处理。

最小模板：

```text
评审来源：
评审时间：
实际访问范围：
未访问 / 未验证范围：
有效反馈压缩：
最强化质疑：
命中点：
处理动作：
主回写位置：
需要同步的次级位置：
暂不采纳项及原因：
是否生成残差 / 版本 / 公开说明：
备注：
```

建议填写说明：

- `实际访问范围` 只记对方真的读到的页面、章节或功能，不替它补读；
- `未访问 / 未验证范围` 要单独列出，防止系统把“没看到”误写成“已证伪”；
- `有效反馈压缩` 最好压成三到五条可执行问题，而不是整段转贴原评审；
- `最强化质疑` 用来把原话改写成真正需要处理的硬问题；
- `命中点` 可选：定义 / 主线 / 边界 / 入口 / 站点交互 / 协作流程；
- `处理动作` 可选：吸收 / 补桥 / 降强 / 转专题 / 登记残差 / 明确拒绝；
- `暂不采纳项及原因` 至少区分三类：对方没读到、命中点不准、当前证据不足；
- 若评审已经触发结构更新，应连带生成版本更新单或公开变更说明。

## 七、活书触发判断单

适用情形：

- 不确定这一轮到底该不该改；
- 评论数量不多，但命中问题很重；
- 热门章节改得太频繁，冷门章节长期不动。

最小模板：

```text
命中章节：
观察时间窗：
本轮评论数量：
重复命中问题：
高权重反馈是否出现：
距离上次修改间隔：
上一版是否已观察充分：
本轮触发方式：
是否进入修改：
若不进入，下一次回看时间：
若进入，必须先生成的对象：
备注：
```

建议填写说明：

- `重复命中问题` 用来判断这是不是同类缺口反复显影；
- `高权重反馈是否出现` 主要看是否直接打在主线、边界、误解点与现实接口上；
- `本轮触发方式` 可选：按期 / 按量 / 按权重 / 三者并行；
- `是否进入修改` 不应只看数量，也要看上一版是否已经给够观察时间；
- `必须先生成的对象` 通常至少包括：快照、修改任务单、必要时的版本单。

## 八、最小使用节奏

如果不想把这几张表用得太重，可以采用一个最小节奏：

1. 先把随手评论压成`评论生长卡`；
2. 先用`评论筛选与冲突归并表`把建设性反馈、灌水和冲突分开；
3. 收到长篇独立评审时，先补一张`独立评审处理单`，再决定要不要拆成反馈表、残差卡或任务单；
4. 每轮准备真的动正文前，先补一张`活书触发判断单`；
5. 一旦决定回写，再先补一张`回写修改任务单`；
6. 遇到不能立即吸收的问题，就补一张`残差登记卡`；
7. 收到值得保留的读者反馈，就压成`反馈回写表`；
8. 一旦正式改动结构，再生成`版本更新单`与必要的公开说明；
9. 每经过一轮较大重写，再统一回看一次这些表之间是否已经形成回路。

这套节奏的目标，不是增加管理动作，而是让重要变化不再只留在聊天记录和个人记忆里。

## 九、这些表之间怎样连接

这几张表不是并列孤立的。更稳的连接方式是：

1. 随手评论先进入`评论生长卡`；
2. 评论过多时，先经过`评论筛选与冲突归并表`；
3. 长篇独立评审先进入`独立评审处理单`，先把已读范围、有效问题和暂不采纳项分开；
4. 真要动正文前，先过一张`活书触发判断单`；
5. 确认要改后，再补`回写修改任务单`锁定范围与上限；
6. 其中当前不能直接吸收的，转入`残差登记卡`；
7. 其中来自读者实际接收结果的，继续压成`反馈回写表`；
8. 一旦正式改动结构，再生成`版本更新单`；
9. 版本更新后，再反过来修正导读、术语和站点入口。

也就是说：

> 评论生长卡负责把碎片带进来，筛选表负责把噪声与冲突分开，独立评审处理单负责把长评审拆成可执行问题，触发表负责决定“现在该不该改”，任务单负责锁定“这次最多改到哪里”，残差负责让暂时吸收不了的问题不丢失，反馈表负责显影实际接收结果，版本单负责让改动留下结构痕迹。

## 十、随手评论生长卡

适用情形：

- 群聊、备忘、语音转写里出现了零散但可能有价值的想法；
- 作者知道这里“像是有东西”，但还远不到正文强度；
- 一段评论里混着误解、补桥、例子、动机和情绪，不宜直接扩写。

最小模板：

```text
原始评论：
压缩表达：
质量判断：
评论类型：
冲突关系：
命中点：
建议修改范围：
改动上限：
建议动作：
建议去向与锚点：
可信状态：
是否转残差 / 反馈 / 版本 / 公开说明：
备注：
```

建议填写说明：

- `原始评论` 尽量保留原句，不急着润色；
- `压缩表达` 用一到两句把它压成可判断的话；
- `质量判断` 可选：建设性 / 待观察 / 灌水 / 恶意风险；
- `评论类型` 可选：术语提醒 / 误解反馈 / 桥接需求 / 现实接口 / 新例子 / 动机方向 / 噪声待观察；
- `冲突关系` 说明它是与哪些评论同向汇聚，还是与哪些评论形成对冲；
- `命中点` 可选：定义 / 主线 / 边界 / 入口 / 专题 / 协作流程；
- `建议修改范围` 尽量写清一句 / 一段 / 小节 / 暂不动正文；
- `改动上限` 可选：措辞修补 / 局部降强 / 局部补桥 / 小节级调整 / 需升级为版本级改写；
- `建议动作` 可选：吸收 / 补桥 / 降强 / 转专题 / 登记残差 / 自然耗散；
- `建议去向与锚点` 标出主书稿、生活卷、专题、AI 协作卷、附录或站点入口，并写清必须带上的摘要、术语表或风格说明；
- `可信状态` 可选：已较清楚 / 猜测 / 材料不足；
- `是否转残差 / 反馈 / 版本 / 公开说明` 用来说明这条评论下一步究竟进入哪套正式结构。

最小投喂提示词：

```text
下面是一些随手评论，请不要直接替我扩写成正文。
请先把它们拆成若干条评论生长卡。
每张卡都要给出：原始评论、压缩表达、质量判断、评论类型、冲突关系、命中点、建议修改范围、改动上限、建议动作、建议去向与锚点、可信状态。
拿不准的地方请明确写“猜测”或“材料不足”。
最后请额外指出：哪些卡值得转残差，哪些卡值得直接形成补桥段落或新专题，哪些卡应该被过滤。
```

## 十一、这一附录的压缩结论

若把这份附录压成一句最实用的话，那就是：

> 凡随手冒出来但可能有价值的，先压成评论生长卡；凡评论太多或彼此矛盾的，先做筛选与归并；凡收到长篇独立评审的，先补独立评审处理单；凡准备真的动正文的，先判断触发条件，再锁定修改边界；凡不能立即并入主线的，就登记残差；凡已经改变结构的，就留版本与公开说明；凡来自读者现实接收的，就再压成可回写反馈，而不是让它们散在对话和记忆里。
