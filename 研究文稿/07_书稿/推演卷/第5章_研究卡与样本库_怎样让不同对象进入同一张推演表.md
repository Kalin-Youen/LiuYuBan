# 第5章 研究卡与样本库：怎样让不同对象进入同一张推演表

## 5.1 本章要解决的问题

前几章已经把差异极限、稳态周期、失稳阈值和样本验证分别推进了一轮。但如果每个对象都只在各自章节里单独讨论，研究很快会重新散开。推演卷还需要一张共同表，让物理样本、学习样本、AI 协作样本和概念组合样本都能进入同一套记录程序。

本章先完成一个最小目标：把“机器自行运行概念组合”纳入样本库。它不是为了让机器直接生成结论，而是检验一件事：

> **概念组合能否先被记录成可复核的生长单元，再由人或高阶模型决定回写、转专题、做工具或登记残差。**

如果这一步能跑通，研究就不再只靠临时灵感推进，而开始具备“机器显影候选，人类裁定方向”的样本生产机制。

## 5.2 统一研究卡格式

当前样本库先采用下面这张最小研究卡：

| 字段 | 含义 |
| --- | --- |
| `sample_id` | 样本编号 |
| `source` | 样本来源，例如脚本运行、读者评论、章节残差、外部案例 |
| `object` | 当前处理对象 |
| `delta` | 识别出的关键差异 |
| `boundary` | 本轮判断边界 |
| `feedback` | 需要接入的反馈或复核方式 |
| `dissipation` | 哪些内容应被筛掉、暂挂或自然耗散 |
| `structure_candidate` | 候选结构 |
| `curriculum_stage` | 适合进入的学习 / 训练阶段 |
| `training_weight` | 当前值得投入的注意权重 |
| `action` | 建议动作：补桥 / 转专题 / 做样本 / 做工具 / 登记残差 |
| `risk` | 风险和不确定点 |
| `decision` | 人或 L4 的裁定 |

这张卡的作用，不是代替正文，而是保证每一次“有想法”都能留下可比较痕迹。

## 5.5 本轮形成的第一批成果

本轮最小成果不是 24 条候选句，而是四个可继续推进的对象。

### 成果一：一张边界检查卡

来源：C0120 `规则 + 因果边界网`

可压成卡片：

```text
若要判断某条规则是否成立，先问：
1. 它在哪个边界内成立？
2. 谁能影响谁？
3. 影响通过什么通道传递？
4. 超出边界后，规则是失效、变形，还是降级为局部经验？
```

这张卡可以进入工具包。

### 成果二：一条 AI 自组织运行链

来源：C0001、C0021、C0066

可压成链条：

```text
候选生成
→ 注意门控
→ 粗粒化成生长单元
→ 规则边界检查
→ 版本回写 / 残差登记
```

这条链可以补回 AI 协作卷第13章，作为 `concept_self_organizer.py` 的理论解释。

### 成果三：一个人类学习样本方向

来源：C0038、C0048

可压成问题：

> 为什么学习者不是缺材料，而是容易卡在某个吸引子式稳定区里？

下一轮可用真实学习任务验证：当学习者总在同一类错误中回返时，是概念边界不清、反馈太弱，还是旧解释框架形成了吸引子。

### 成果四：一个脚本改进项

来源：C0037、C0035、C0057、C0065

当前脚本已经能给出“跨层较大，需先补中间先修概念”的判断，但还没有自动补出中间桥。下一轮应新增：

```text
missing_prerequisites:
- 需要先补的概念
- 为什么缺这一步
- 补完后再组合的建议顺序
```

## 5.6 本轮的失败条件与限制

本轮不能夸大为“机器已经能自动组织理论”。限制很清楚：

1. 本轮没有启用模型深评，只是离线启发式筛选；
2. 候选句仍有模板味，不能直接进入正文；
3. `training_weight` 仍来自启发式估计，不是从真实回写成功率中学习出来；
4. 知识图谱节点的阶段和难度仍较粗，需要后续人工标注或样本反推；
5. 当前只跑了一轮，不能判断稳定性。

因此，本轮最稳的结论是：

> **概念自组织系统已经能生成可复核候选，并能初步分出工具、补桥、样本和残差；但它还没有形成自动回写能力，更没有资格替代人工裁定。**

## 5.7 下一轮执行任务

下一轮不应继续盲目增加候选数量，而应做四件事：

1. 给脚本增加 `missing_prerequisites` 字段；
2. 启用一次模型深评，比较启发式评分和模型复核是否一致；
3. 把 C0120 做成正式工具卡；
4. 把 C0001 / C0021 / C0066 补回 AI 协作卷第13章，形成“机器组合系统”的稳定协议。

本章的第一轮结论可以压成一句：

> **让研究跑起来，不是让机器直接写结论，而是让机器稳定生产可检查候选，再让样本、复核和回写决定哪些候选真正长成成果。**

## 5.8 第二轮：启用模型深评后的运行约束

随后启用本地 NVIDIA 兼容 API 做了一轮模型深评。运行发现一个直接约束：`12` 条候选会触发读取超时，而 `6` 条候选可以稳定返回。因此，当前运行形式不宜追求单轮大批量，而应采用：

```text
小批量生成
→ 模型深评
→ 人工抽样复核
→ 合并多轮结果
→ 再进入回写或残差登记
```

第二轮成功运行命令为：

```powershell
python scripts\concept_self_organizer.py --count 6 --seed 20260514 --use-ai --output-dir .collaboration\concept_runs
```

本轮模型返回了 5 条深评结果，另有 1 条未返回。这说明模型深评本身也需要被当作不完整反馈，而不是绝对裁定。脚本已据此改为：若启用 AI 但某条候选未被返回，则在 Markdown 中标记为 `unreviewed`。

### 5.8.1 模型深评确认的候选去向

| 编号 | 组合 | AI 裁定 | AI 去向 | 可采取动作 |
| --- | --- | --- | --- | --- |
| C0001 | 约束继承 + 预测误差 | keep | 主书 | 先检查主书是否已有重复，再决定是否补桥 |
| C0004 | 落差 + 吸引子与稳定区网 | keep | 残差 | 登记为“落差如何进入吸引子解释”的残差 |
| C0007 | 耗散 + 阶梯式学习 | keep | 专题 | 可扩成“学习中的耗散与阶段设计”专题 |
| C0008 | 约束继承 + 光速、量子与物质母网 | keep | 工具 | 可做成“物理章节先修顺序检查卡” |
| C0014 | 阶梯式学习 + 非均匀起源网 | keep | 残差 | 暂记为“非均匀起源如何教学排序”的残差 |

### 5.8.2 本轮对运转形式的优化判断

1. **批量上限**：当前 API 深评以 `6` 条为稳妥批量；超过后应拆分。
2. **深评不是全覆盖**：模型可能漏评候选，系统必须显式标记 `unreviewed`。
3. **AI 去向与脚本建议不总一致**：例如脚本建议“做样本”，模型可能建议“工具”。这类分歧本身就是复核材料。
4. **成果不应直接回写正文**：本轮更适合先形成工具卡、残差卡和专题候选，再由 L4 裁定。

这一轮之后，系统的运行形式可以从“单次生成结果”升级为“多轮小批量候选池”。下一步应增加一个合并器，把多轮 `.collaboration/concept_runs/*.json` 汇总成总表，统计哪些概念组合反复出现、哪些去向最稳定、哪些一直被模型标为高风险。

## 5.9 第三轮：建立可视化监督控制台

为了让“研究跑起来”不只是命令行动作，本轮进一步补出一个本地可视化控制台：

```powershell
python scripts\concept_console_server.py --port 8876
```

控制台地址：

```text
http://127.0.0.1:8876
```

它当前提供四个能力：

1. **查看历史批次**：读取 `.collaboration/concept_runs/*.json`，展示每轮候选、AI 裁定、AI 去向、风险和缺失先修。
2. **一键运行**：通过 `/api/run-once` 调用 `concept_self_organizer.py`，生成新的概念组合批次。
3. **循环运行**：通过 `/api/start-loop` 按设定间隔自动运行；通过 `/api/stop-loop` 停止。
4. **监督状态**：显示循环是否运行、最近批次数、候选数、深评数和错误状态。

本轮已经验证：控制台可以成功读取 `7` 个历史批次，并通过 API 触发一轮 `3` 条候选的离线运行，运行后批次数增加到 `8`。这说明系统已经从“脚本工具”推进到“可监督运行界面”。

当前控制台仍然只是本地研究台，不是公开站点功能。它不应直接部署到 GitHub Pages，也不应暴露 API key。API key 仍只放在本地忽略文件 `.concept_ai.local.json` 中。

下一步优化方向应是：

1. 给控制台增加“合并多轮结果”的总表；
2. 统计每个概念的出现频率、平均权重、常见 AI 去向和高风险比例；
3. 增加“标记为已处理 / 待回写 / 已登记残差”的人工裁定字段；
4. 把人工裁定再反馈给下一轮权重更新。

到这一步，概念自组织系统已经具备了一个最小闭环：

```text
脚本生成候选
→ 模型深评
→ 控制台监督
→ 人工裁定
→ 回写或残差登记
→ 下一轮权重调整
```
## 5.10 有用还是只是玩：首轮聚合判据

如果这个系统只能不断生成“看起来有意思”的句子，它就是玩具；只有当它能把多轮输出压成可统计、可复核、可回写的对象时，才算进入研究工具阶段。

本轮新增了 `/api/aggregate` 聚合接口，并把聚合结果接入本地控制台。聚合不再只看某一条候选是否顺眼，而是统计：

1. 总批次数、候选数、已复核数、`keep` 数和未复核数；
2. AI 去向分布：主书、专题、工具、残差等；
3. 高频概念和重复组合；
4. 可回写候选；
5. 高风险候选；
6. 当前系统更接近“研究工具”还是“玩具”的判定。

截至 2026-05-12 17:24，系统共积累 `13` 批、`101` 个候选，其中 `34` 个经过模型复核，`33` 个为 `keep`，`1` 个为 `defer`，`8` 个为 `unreviewed`。去向分布为：主书 `12`、专题 `8`、工具 `8`、残差 `5`。

这组数据不能说明机器已经会自动完成研究，但它说明三件事：

1. **它不是纯玩具**：输出已经能被跨批次聚合，并能稳定分出主书、专题、工具、残差等去向；
2. **它还不是自动研究者**：`keep` 率过高，说明模型复核仍偏宽，必须增加人工裁定和反例样本；
3. **它最有用的位置是候选生成与排序**：机器负责生成组合、暴露桥接点、记录风险；人负责决定哪些进入正文、哪些变成工具卡、哪些登记为残差。

因此，本轮最稳的判断是：

> 概念自组织系统已经有研究用途，但它的用途不是替人下结论，而是把“可能值得研究的概念连接”批量生产出来，并通过聚合指标帮助人更快裁定。

### 5.10.1 有用条件

系统满足下列任一条件，就应继续投入：

1. 产生可回写候选；
2. 暴露缺失先修、边界风险或概念桥接点；
3. 多轮后出现重复概念、重复组合或稳定去向；
4. 人工裁定结果能反过来影响下一轮权重。

### 5.10.2 玩具条件

系统若长期满足下列条件，就应降级或停止：

1. 输出不能被聚合；
2. 没有候选通过复核；
3. 没有重复模式或稳定去向；
4. 没有任何候选能进入正文、专题、工具卡或残差登记；
5. 人工裁定无法反馈到下一轮权重，只是反复看新句子。

### 5.10.3 下一步真正要补的环节

现在最大残差不是“再多生成一些”，而是让人工裁定能回流。下一轮应补三件事：

1. 在控制台给候选增加人工状态：`待处理`、`已回写`、`已登记残差`、`拒绝`；
2. 保存人工裁定为本地文件，不进 Git，不暴露 API key；
3. 下一轮生成时读取人工裁定，提高被回写类型的权重，降低被拒绝类型的权重。

这样系统才会从“会跑”继续前进到“会从裁定中学习”。
