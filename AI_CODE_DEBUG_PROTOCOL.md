# AI 代码调试强约束协议

这份协议用于约束 AI 在代码修复中使用差结构学习法。它的目的不是让 AI 说出更漂亮的分析，而是阻止它在同一类 bug 上反复叠补丁。

## 触发条件

满足任一条件时，必须启用本协议：

- 同类 bug 已经修过两次仍失败。
- 用户反馈“还是一样”“更糟了”“你在打转”“换个思路”。
- 症状在性能、崩溃、状态残留、入口差异、撤销重做、缓存、UI 状态之间迁移。
- AI 已经能说出“入口差异、生命周期、边界”，但代码仍没有稳定修好。

## 禁止动作

触发后，禁止直接继续：

- 追加一个局部 `if`。
- 包一层 `try/except`。
- 在函数末尾随手补一次状态切换。
- 只修最新一次看到的症状。
- 用“恢复状态”“加强判断”“增加兼容”这类话代替可执行判准。

## 动代码前必须产出的四件事

```text
比较对象：
本次 bug 的入口差异是什么？

状态所有者：
哪个状态是用户/文档/会话拥有的？
哪个状态只是过程临时占用的？
哪个函数或模块有权写它？

最终判准：
改完后用什么具体事实证明成功？
必须写成可执行事实。

失败条件：
什么观察说明这次修复无效？
```

## 生命周期图

反复失败时必须画出：

```text
进入前状态:
操作中临时状态:
finally/cleanup 状态:
最终强制状态:
返回后状态所有者:
```

最终强制状态必须来自用户目标和成功判准，不能来自当前函数末尾刚好方便设置的状态。

## 差异轴到代码位置

每个差异轴必须落到代码位置：

- 入口差异 -> 哪个 `invoke` / `execute` / UI draw / command / test 初始化不同？
- 生命周期差异 -> 哪个 `try/finally`、上下文管理器、事务、回调或异步任务持有状态？
- 边界差异 -> 哪个函数负责读写该状态？哪个模块不应该写？
- 所有权差异 -> 哪个状态属于用户/文档/会话？哪个只是过程临时占用？
- 反馈差异 -> 哪个日志、断言、测试或状态探针能证明它真的生效？
- 判准差异 -> 哪个可执行检查决定“完成”？

## 破局动作

如果同一方向已经失败，必须至少选择一个换层级动作：

- 从补症状转向改状态所有权。
- 从函数末尾补救转向完整生命周期协议。
- 从单入口修复转向统一入口初始化。
- 从保留对象引用转向按需重新解析。
- 从抽象解释转向可执行 invariant。
- 从猜测原因转向加最小探针验证真实状态。

如果没有破局动作，必须停止修改并说明缺少哪条事实。

## 最终交付格式

```text
反复失败模式：是/否
比较对象：
状态所有者：
生命周期图：
最终判准：
失败条件：
代码落点：
破局动作：
最小修改：
验证结果：
残留风险：
```
