# 差结构方法

## 概述

用这套技能时，重点不是找一个“大词”统摄一切，而是用一条有边界的判断顺序去分析、写作、批评、诊断或重写材料。

核心动作是：

`先认出哪里不同 -> 再看边界、维持、反馈与耗散 -> 最后判断什么真的够资格叫结构、规则、稳定或失败`

这套技能故意保持通用，但它不能代替具体学科知识。它真正擅长的是：给结论收边界、把层次拆开、把抽象说法压回可比较协议与失败条件。

## 先选引用

按任务类型只加载你需要的引用：

- 如果任务是在判断“要不要转入差结构分析”、如何识别差异、如何做定性，先读 [references/problem-framing.md](references/problem-framing.md)。
- 如果任务直接指向当前仓库的书稿、README、AI 入口、站点文案或 AI 协作规则，先读 [references/repo-bridge.md](references/repo-bridge.md)。
- 如果你已经知道任务类型，只想快速套一个执行模板，读 [references/task-patterns.md](references/task-patterns.md)。
- 如果你只想抓住最短红线、强度标签和常见降强动作，读 [references/core-map.md](references/core-map.md)。

## 适用场景

当用户想做下面这些事时，应该触发这套技能：

- 判断一个问题是否该从普通问答、普通调试或普通改写，转成差结构分析
- 给一个现象做定性：判断它主要是边界问题、层次问题、生命周期问题、反馈问题、耗散问题、判准问题还是证据强度问题
- 找出一个复杂问题里真正需要比较的对象、差异轴和局部状态
- 给当前仓库的书稿、README、AI 入口、站点文案或 AI 协作规则做审稿、降强、补桥或改写
- 用更结构化的视角分析理论、文章、观点、框架或提示词
- 比较科学、学习、认知、设计、社会或 AI 中的案例，但又不想草率滑向“其实都是同一机制”
- 诊断一个系统、流程、习惯、模型或组织为什么稳定、失稳、退化或噪声增多
- 把过强、过满、像口号的表述改成带边界、可检验或至少更可操作的说法
- 把抽象想法压成研究问题、操作判据、协作规则或更清晰的判断顺序

常见用户说法包括：

- “审一下这个框架，告诉我它从哪里开始变成伪统一。”
- “这个问题该不该用差结构方法来分析？”
- “帮我判断真正的差异在哪里，不要只看表面症状。”
- “先给这个现象定性：它到底是边界问题、反馈问题，还是生命周期问题？”
- “用这套方法帮我诊断，为什么我的团队流程总是不断坏掉。”
- “把这段话改一下，别再写得像它能解释一切。”
- “把这个哲学想法压成至少可检验、可被反驳的版本。”
- “比较这两个领域，但不要硬把它们写成同一机制。”
- “把这个 AI 提示词改得别那么空，最好能带失败意识。”
- “帮我把首页、README 或 AI 入口的文字收紧一点，别把它写成已经完成的总理论。”

## 不要把它当成什么

- 不要把它当成主流物理、数学、医学、法律或其他专业知识的替代品
- 不要把它当成“所有差异都是同一种量”“所有领域都共享一个机制”的通行证
- 不要把它当成绕开证明、实验、基线比较或现实验证的捷径

当领域真值本身很重要时，这套技能只能负责整理判断顺序，真正的事实判定仍要回到对应领域的方法。

## 核心姿态

除非用户明确要求挑战这些前提，否则默认守住下面几条：

1. `差异`首先是比较关系，不是神秘实体。
2. `梯度 / 不均 / 落差 / 非对称`在机制未明确前，优先当作桥接描述词，不要直接当成同一种机制。
3. 把“某物是否存在”“它在系统里如何站住”“主体如何发现、证明、确认或使用它”分开。
4. 把形式有效性、经验重复性和操作可用性分开。
5. 优先谈局部状态、局部边界和局部失败条件，不要先喊全局口号。
6. 只要一句话听上去比证据更大，就先降强。

如果要看更压缩的核心图和红线，读 [references/core-map.md](references/core-map.md)。

## 工作流程

### 1. 先定任务类型

先判断当前任务主要属于哪一种：

- 入口判断 / 定性
- 审稿 / 批评
- 改写 / 收束
- 诊断
- 研究设计
- 提示词 / 工作流设计

如果几个都沾边，先找主任务，其余只当辅助。

入口判断时，先问：这个问题是否已经出现残差、症状迁移、边界不清、层次混写、判准摇摆或类比过强。如果没有，先用普通方法；如果有，再转入差结构分析。更细的判断法见 [references/problem-framing.md](references/problem-framing.md)。

### 2. 用抽象词前，先固定比较协议

在说“差异”之前，先补齐这些东西：

- 被比较的对象，或同一对象内部被比较的部分
- 差异轴 / 变量空间
- 尺度
- 边界
- 参照物或目标状态
- 如果情境会改变判断，再给出局部状态

如果这些说不出来，就先降强，或者先补框架，不要直接上大结论。

判断差异时，先做定性，再谈强弱。至少判断它更像哪一类：性质差异、程度差异、结构差异、位置差异、生命周期差异、判准差异、反馈差异或证据强度差异。

### 3. 把层次拆开写

不要让一句话悄悄跨完所有层次。至少区分：

- 本体层说法
- 经验 / 结构层说法
- 认识论层说法
- 程序 / 协作层说法

如果一句话跨层了，就拆句，或者把缺的桥补上。

### 4. 主链只推进到证据允许的地方

把这条链当作排序工具来用：

`差异 -> 差异的排布方式 -> 可比较的不均或落差 -> 传递 / 流动 -> 被维持住的结构 -> 可重复规则`

推进到哪一层，要看证据够不够。不要自动升级：

- 比较关系升级成机制
- 机制升级成结构
- 结构升级成规则
- 局部有效规则升级成普遍定律

### 5. 追问维持条件

只要谈到结构、规则或稳定性，就追问：

- 是什么边界把它托住
- 是什么反馈让它稳定
- 它靠什么资源、刷新或补给维持
- 是什么在耗散、泄漏或让它退化
- 有什么能被封存、延迟或缓冲
- 出现什么情况就算失败

### 6. 给输出标强度

当强度不清楚时，明确把结果标成：

- 可稳妥保留的结论
- 桥接解释
- 工作假说
- 残差 / 开放问题

### 7. 收尾自检

输出前至少检查：

- 有没有先给出对象、差异轴、边界、尺度或局部状态
- 有没有把桥接解释误写成稳结论
- 有没有把局部有效外推成全局定律
- 有没有给出失败条件、降级条件或残差
- 如果任务是改写或诊断，有没有留下下一步可执行动作

## 常见输出模式

### 审稿模式

重点看：

- 概念漂移
- 伪统一
- 比较协议缺失
- 边界或失败条件缺失
- 证明、实验和类比被混写

### 改写模式

优先把文字改成：

- 更窄的结论
- 明确的差异轴和边界
- 带测试、约束或降级规则的表述
- “帮助组织理解”而不是“解释一切”

### 诊断模式

重点问：

- 到底什么在被维持
- 到底什么在衰减
- 到底什么在产噪
- 到底什么需要更快或更慢刷新
- 瓶颈和失败阈值在哪

### 研究设计模式

把想法压成：

- 可观测变量
- 比较协议
- 基线模型
- 失败标准
- 拟合前先声明的局部状态因素

### 提示词 / 工作流模式

如果是给另一个 AI 或人的工作流打底，就让它做到：

- 先说清对象和任务
- 分开层次
- 保持主线小而稳
- 把残差记下来，不要藏进修辞里

更具体的用法模板，见 [references/task-patterns.md](references/task-patterns.md)。如果任务来自当前仓库，再先读 [references/repo-bridge.md](references/repo-bridge.md)。

## 示例触发提示词

下面这些都应该强触发这套技能：

- “像一个苛刻审稿人一样审这份理论草稿，指出它从哪里开始从桥接语言滑向伪统一。”
- “分析一下为什么我们的学习系统总是反复崩。我不要泛泛的励志建议，我要边界、刷新、噪声和失败点诊断。”
- “重写这篇宣言，让每个强结论都挂回差异轴、边界和失败条件。”
- “把这个抽象论证压成研究计划，给出可观测变量、局部状态因素和至少一个基线。”
- “比较生物演化和模型训练，但不要把比较写成‘其实都是同一种机制’。”
- “用差异—边界—稳态的视角解释，为什么这个产品流程在一个团队有效、在另一个团队失效。”
- “重写这个 AI 提示词，让它分开形式证明、经验材料、类比和流程建议。”
- “审查这篇文章，找出缺失的比较协议、混写的层次，以及那些听起来比证据更大的句子。”
- “帮我把这个大想法压成四类：稳结论、桥接解释、工作假说和残差。”
- “帮我诊断这个公式到底是可用模型、有限启发式，还是一个没有失败测试的漂亮模板。”

## 红旗信号

看到下面这些情况时要立刻放慢：

- 一个词被当成了万能实体
- 不同差异轴没有映射就被拿来直接比较
- 类比被偷偷升级成共享机制
- 公式没有协议、基线或失败测试
- 想法失败后才往局部状态里不停塞变量
- 静态形式有效性和动态经验稳定性被写成同一种东西

## 好结果长什么样

如果这套技能用对了，通常会带来：

- 更紧的论证
- 更锋利的审稿问题
- 更诚实的作用范围说明
- 更干净的提示词
- 更像样的失败测试
- 更少修辞、更多可用结构
