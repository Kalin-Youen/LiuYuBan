# 仓库桥接

当任务直接指向当前仓库的书稿、专题、`README.md`、`AI_ENTRY.md`、站点首页文案、AI 协作规则或评论入口设计时，先读这个文件。

## 1. 先守住当前稳定定位

默认把“差结构学习法”写成：

- 一套正在生长的判断方法、学习基础设施或协作方法
- 不是已经定型的总本体论
- 不是“一个词解释一切”的公共口号

除非用户明确要求拔高，否则优先降强到这个强度。

## 2. 优先级顺序

仓库内任务默认按这个顺序取舍：

1. 入口清晰
2. 边界清晰
3. 层次分开
4. 失败条件和残差可见
5. 再谈更大的统一语言

如果更大的口号会伤害前四项，就先放弃口号。

## 3. 仓库内的常见安全改写

优先这样写：

- “这是一套完成的总理论” -> “这是一套正在生长的判断方法或公共方法基础设施”
- “X 证明了所有领域共享同一机制” -> “X 提供了一条可比较的判断顺序”
- “差异生成一切结构” -> “差异帮助识别结构，结构是否成立仍要看维持条件与失败测试”
- “AI 已经验证这个理论” -> “AI 提供了反馈、压力测试或改写样本，不等于独立证明”

## 4. 审仓库材料时优先抓什么

- 五卷分工有没有被混写
- `差异 / 差结构 / 落差 / 流 / 结构 / 规则` 有没有被偷写成同义词
- 桥接解释有没有被提前写成稳结论
- 站点入口是不是先帮人进入，而不是先要求接受总理论
- AI 协作规则是不是把“未读到”、残差、版本与回写说清楚

## 5. 推荐输出形状

如果是审稿或改写当前仓库材料，优先按这个顺序输出：

1. 可稳妥保留的部分
2. 需要降强的部分
3. 需要补桥的部分
4. 建议改写
5. 残差或暂不该说满的地方

## 6. 需要时再加载哪些文件

- 项目定位与机器入口：`AI_ENTRY.md`
- 仓库结构与五卷一站：`README.md`
- 如果任务只涉及某一卷，再去读对应卷的原文
- 如果未读到某页或某数据，明确说 `未读到`
